## 1. 프로젝트 개요

- 1인 개발 중인 한국어 텍스트 RPG, 포스트아포칼립스 사이버펑크 세계관 "데드존(Dead-Zone)".
- 리포: https://github.com/rorena15/T_RPG
- 메인 실행 파일: `개발/Main.py`. 보조: `개발/db_init.py`(SQLite 스키마+장비 데이터), `개발/sys_log.py`(로깅/텔레메트리), `database.json`(서사/세션/소모품 데이터), `master_formulas.json`(연산 공식 SSOT).
- 현재 구현 범위: 4막 중 **1막(낙인)** 만 완성. 2~4막은 시놉시스 수준(`스토리/본편 스토리- 프로토콜 리부트.md`).
- 본편 완성 시 예상 분량: 순수 스크립트 8만~10만 자, 1회차 6~8시간. (문서에 이미 명시되어 있던 수치)

## 2. 이번 세션에서 한 개발 작업 (시간순)

### 2.1 장비 데이터 통합 (db_init.py)
- `장비/` 폴더의 5개 등급별 MD 파일(급조/규격/정제/기업제/유물, 총 462종)을 전부 파싱해 `db_init.py`의 `equipment` 테이블에 통합.
- ID 네이밍: `{슬롯코드}_{등급코드}_{일련번호}` (예: `WEAPON_STD_01`, `RING_LEGACY_05`).
- 슬롯: main_weapon, cyberdeck, cybernetic_parts, back_gear, face, top, bottom, footwear, necklace, ring, custom_part (11종, 슬롯당 42개 + 기본무기 1개 = 463행).
- `equipment` 테이블에 `slot`, `slot_weight`(매트릭스 엔진 문서 기준 가중치 1.5/1.2/1.0/0.5) 컬럼 신설.
- `database.json`의 `SESSIONS_DB` 보상 ID 중 옛 발췌 체계(`ARMOR_SCRAP_01` 등)를 새 슬롯 기반 ID로 교정.

### 2.2 `master_formulas.json` KeyError 버그 수정
- 원인: `Main.py` 524번째 줄이 `MASTER_FORMULAS["formulas"]["reputation_multiplier"]["divisor"]`를 참조했는데, 실제 JSON 키는 `reputation_factor`이고 `divisor` 같은 숫자 필드 자체가 없었음(공식이 문자열로만 저장됨).
- 수정: 분모 2000을 직접 사용하도록 변경 (`f_multiplier = 1.0 + (player.reputation / 2000) * 1`). 공식 문서의 `F = 1 + (R/2000) * region_flag`와 동일.

### 2.3 전투 UX 정리 (wait_for_keypress 잔재 제거)
- 본인이 이미 `combat_loop` 내 대부분의 `wait_for_keypress()`(키 입력 강제 대기)를 `time.sleep(N)`(자동 진행)으로 리팩토링해둔 상태였음.
- 딱 2곳(후퇴 중 피격 분기, 잘못된 명령어 입력 분기)에 잔재가 남아 흐름이 끊기던 것을 `time.sleep(1)`로 통일.
- 전투 결과 화면(사망/후퇴완료/승리) 3곳의 `wait_for_keypress()`는 의도적으로 유지(결과를 충분히 보고 넘어가게).

### 2.4 `track` 데코레이터 고도화 (플레이 데이터 수집)
- `sys_log.py`의 `track` 데코레이터를 SQLite `events` 테이블에 적재하도록 재작성.
- 수집 항목: session_id(UUID, 프로세스당 1개), timestamp, func_name, args_json, kwargs_json, result_json, duration_ms, success, error_type, error_message, player_hp, player_turn_context.
- Player 같은 큰 객체는 `to_dict()` 자동 인식해 핵심 지표만 축약 저장, HP는 별도 컬럼으로 분리(SQL 필터링 용이).
- 텔레메트리 적재 실패가 게임 로직의 예외 전파를 절대 가리지 않도록 try/except로 완전 격리 (실제 예외 발생 시나리오로 검증함).
- `db_init.py`의 DB 초기화 로직 변경: 기존엔 부팅마다 `stigma_data.db` 파일 자체를 삭제했는데, 이러면 `events`에 누적된 플레이 데이터가 매번 날아감. → `equipment`/`consumables` 테이블만 DROP 후 재생성, `events`는 보존하도록 수정.

### 2.5 유물(0등급) 장비 테스트 지급 제거 + 히든 커맨드
- `Player.__init__`에서 `WEAPON_LEGACY_01` 자동 지급(테스트용 잔재) 제거. 정식 빌드는 빈 인벤토리로 시작.
- `manage_inventory()` 메뉴에 **`DEV_GRANT_LEGACY`** 히든 커맨드 추가. 메뉴 목록(1~4, 0)에는 노출 안 되고, 정확히 이 문자열을 직접 입력해야만 0등급 유물 66종 전체 지급. 개발자 검증용.

### 2.6 1막 데모 보상 등급 확장
- 1막 5개 세션(`database.json`의 `SESSIONS_DB`) 각각에서 선택지 1개씩, 총 5곳을 4등급(급조)에서 3등급(규격)으로 교체/신설:
  - 세션1 완력 해체 → `WEAPON_STD_01`
  - 세션2 정제 펌프 급조 → `NECK_STD_01`
  - 세션3 패킷 주입(해킹) → `DECK_STD_01`
  - 세션4 웜 심기(해킹) → `RING_STD_03`
  - 세션5 완력으로 문 개방 → `CYBER_STD_01` (원래 보상 없던 선택지에 신규 부여)
- 보상 없는 자원/트레이드오프형 선택지는 건드리지 않음.

### 2.7 턴 기반 적 스탯 스케일링 (파밍 중요도 강화)
- **목적**: 파밍(장비 강화) 없이 초반 장비로 끝까지 버티는 게 가능했던 문제 해결. 턴이 지날수록 적이 강해지게 만들어 파밍을 필수로 만듦.
- **설계 합의 과정**: "장비가 좋아지면 적도 강해진다"는 역설계는 개연성이 떨어진다는 지적(본인) → 턴수를 메인 드라이버로, 플레이어 체력은 위험할 때만 완화하는 안전핀 역할로만 쓰기로 합의.
- `Player.turn_count`: `consume_resources()`(이동/탐색) 1회당 +1. 저장/로드 보존, 구버전 세이브 호환(키 없으면 0).
- `Player.difficulty`: 새 게임 시작 시 EASY/NORMAL/HARD 3종 선택. 로드 게임은 저장된 값 유지.
- `DIFFICULTY_SCALING_RATE = {"easy": 0.01, "normal": 0.02, "hard": 0.035}` — normal 기준 50턴에 적 hp/atk 2배.
- `get_turn_scale_multiplier(player)`: `1.0 + turn_count * rate`. 단, 전투 시작 시점 HP가 30% 미만이면 그 전투에 한해 증가분을 절반으로 완화(`LOW_HP_RELIEF_THRESHOLD=0.3`, `LOW_HP_RELIEF_FACTOR=0.5`).
- `combat_loop`의 고정 atk/hp에 스케일 적용. 탈출 후 재조우 시 이미 스케일링된 저장 hp를 재곱셈하지 않도록 이중 스케일링 방지 처리됨.
- 상태창(`show_status`)에 `[진행 턴] [난이도] [적 위협 배율]` 노출 추가 — 왜 어려워지는지 플레이어가 인지 가능하게.

## 3. 서사/기획 관련 논의 요약 (코드 변경 없음, 의사결정 기록용)

- **분량 현황**: 1막만 완성(약 1.7만 자), 전체 본편의 약 1/5 분량. 2~4막은 시놉시스만 있어 신규 집필 3~4배 더 필요.
- **2~4막 신규 시스템 필요 항목** (시놉시스에만 있고 엔진엔 미구현):
  - 3막: 0등급 유물 연성(+10 공명 정제), "신격 스케일링", 십만 배수 대미지 — `apply_dynamic_scaling`이 이 정도 스케일을 감당하게 추가 설계 필요.
  - 4막: 연산 오염 스택(Stack≥100 시 RAM 0 영구 고정), 포맷/동기화 양면 분기 엔딩 — `run_ending()` 쪽 신규 분기 필요.
  - 그 외(2막, 일반 세션 확장)는 엔진 그대로 두고 `SESSIONS_DB` 데이터만 늘리면 됨.
- **LLM(제미나이 등) 활용 집필 의견**: 백지 집필보다 "1막 패턴 복제 + 시놉시스 살붙이기"가 효율적. 단, 막 단위로 끊어서 생성하고 떡밥/변수 정합성은 사람이 검수하는 반자동 워크플로우 권장. 한 번에 전체 통으로 생성하면 톤 일관성과 떡밥 회수가 어긋날 위험.
- **공개 플랫폼**: itch.io(언리스티드 링크)가 1순위, 추후 반응 보고 스팀 무료 데모 고려. 국내는 디스코드 "인디게임 한국"류 커뮤니티, 루리웹/인벤 인디 게시판 추천.
- **UI**: 풀 그래픽 전환은 1인 개발에 과함(풀타임 6개월~1년+ 체감). 절충안은 일러스트+카드 UI 하이브리드. 최소 마지노선은 exe 더블클릭 실행 + 폰트/색상 정리, 또는 웹 포팅.
- **역공학/치트 방지**: PyArmor는 소스코드(.py) 난독화는 가능하지만 `stigma_data.db`/`stigma_save.json`/`master_formulas.json`/`database.json` 같은 런타임 데이터 파일은 보호 못 함(별개 문제). 싱글플레이 텍스트 RPG 특성상 세이브 치트 방지에 본격 투자할 우선순위는 낮다고 판단(타인에게 영향 없는 자기 플레이 영역).
- **재미 평가**: 세계관/성장 시스템(행위 가중치 누적 → 자동 전직)은 탄탄하나, 전투 의사결정 깊이(4선택지 단조 반복)와 선택지 트레이드오프(보상이 사실상 동치인 경우 많음)가 약하다는 우려 제기. 직접 플레이테스트 필요성 강조 — 지인 부재 문제는 itch.io 언리스티드 + 인디 커뮤니티 품앗이로 해결 권장.

## 4. 알아두면 좋은 코드 구조 메모

- `Player.to_dict()`/`from_dict()`가 세이브/로드와 텔레메트리 직렬화 양쪽에서 재사용됨. 새 필드 추가 시 양쪽 다 업데이트 필요, 구버전 세이브 호환을 위해 `from_dict`는 항상 `.get(key, 기본값)` 패턴 유지할 것.
- `apply_dynamic_scaling(raw_dmg, raw_hp, highest_equip_tier)`는 **표시용 연출 스케일링**(장착 장비 등급에 따라 화면에 보이는 숫자만 부풀림)이지 실제 연산값에는 영향 없음. 적 실제 스탯(turn_count 기반)과는 별개 시스템이니 혼동 주의.
- `get_equipment_data(item_id)`는 SQLite `equipment` 테이블 조회, 없으면 fallback 더미값 반환.
- `database.json`(서사/세션)과 `stigma_data.db`(장비/소모품/이벤트)는 역할이 분리되어 있음. 장비 추가는 `db_init.py`, 세션/스토리 추가는 `database.json`.
- 작업 시 주의: 이번 세션 중 `개발/` 폴더 안에 `database.json` 구버전 사본이 남아있어서 테스트 결과가 한 번 꼬인 적 있음. 루트의 `database.json`이 항상 최신 원본이므로, 작업 전 `개발/` 폴더 사본을 덮어쓰고 시작할 것.
